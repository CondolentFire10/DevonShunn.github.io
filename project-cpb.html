// THE PROBLEM I FOUND

A System That Was Slowing Everyone Down

As Director of Experiential Programming at Claremont McKenna's CPB, I plan five events per semester — hikes, off-campus experiences, skill workshops — all substance-free and designed to give students something genuinely new to try. With a $3,500 per-semester budget and real events on the line, the logistics matter.

The system everyone was using to handle signups was a Google Form emailed out each week. Students would submit the form, then directors would manually read every response, individually message each accepted student, individually message five people on a waitlist, and manually track everything in a spreadsheet. Every week. For every event. For every director across the entire CPB.

The problems compounded quickly. Some events were raffles, not first-come-first-served — and DOS certified drivers needed priority in both systems, neither of which the form could handle. There was no automatic waitlist promotion when someone cancelled. No way to track who consistently no-showed. No confirmation emails. Just a spreadsheet and a lot of manual DMs.

I decided to build a replacement instead of working around it.

Every Feature, Explained

Built around the actual requirements of CPB events — not a generic signup form. Each piece exists because the old system couldn't do it.

CMC-Only Accounts

Registration is locked to @cmc.edu email addresses. Students store their DOS certification and whether they have a car and are willing to drive with reimbursement. All of this is editable from their own account page at any time.

First Come First Served

Standard FCFS with automatic waitlisting. When the event fills, new signups go to the waitlist in order. If DOS priority is on, certified drivers bypass the capacity limit — they always get a confirmed spot regardless of how full the event is.

🎟️

Raffle Events

Directors set events as raffles with a close time. Students enter the pool. The director draws with one button — winners are selected randomly, with DOS drivers counted twice if DOS priority is on. Every entrant is emailed their result automatically.

Automatic Email System

Every action sends the right email: confirmation, waitlist position, promotion when a spot opens, raffle win or loss. Every email has a one-click cancel link with a unique token — no login required to cancel, and the token can't be reused.

Gmail + NodemailerSecure cancel tokenNo login to cancel
⚠️

Late Cancellation Tracking

Cancelling within 24 hours of an event counts as a late cancellation — unless the person was promoted from the waitlist within that same window (so they can't be penalized for something they didn't cause). Email warning at 1 strike, final warning at 2, suspended at 3.

🚫

No-Show Tracking

Directors submit an attendance roster after each event. Anyone who had a confirmed spot, didn't attend, and didn't cancel gets a no-show strike. Email warning on the first, suspended on the second — a harder threshold than late cancels because it's a worse outcome for everyone.

// HOW IT'S BUILT

The Technical Stack

No frameworks, no build step, no unnecessary dependencies. The whole thing runs from a single Node.js process and a SQLite file — deployable anywhere Node runs.

🟢

Node.js + Express

Web server and all routing. Every page is server-rendered HTML — no client framework needed.

🗄️

SQLite

Zero-config database stored as a single file. Handles users, events, signups, and strike log.

📧

Nodemailer

All emails sent via Gmail App Password. Dev mode logs to console so the app works without credentials.

🔑

JWT + bcrypt

Passwords hashed with bcrypt. Sessions stored as HTTP-only cookies signed with JWT tokens.

server.js — automatic waitlist promotion 
// When someone cancels, find #1 on the waitlist and promote them automatically
async function promoteWaitlist(eventId) {
  const next = db.prepare(`
    SELECT s.*, u.email, u.name FROM signups s
    JOIN users u ON u.id = s.user_id
    WHERE s.event_id = ? AND s.status = 'waitlist'
    ORDER BY s.position ASC LIMIT 1
  `).get(eventId);

  if (!next) return; // nobody waiting — nothing to do

  const newToken = crypto.randomBytes(28).toString('hex');
  db.prepare("UPDATE signups SET status='accepted', cancel_token=?, accepted_at=? WHERE id=?")
    .run(newToken, new Date().toISOString(), next.id);

  // Shift everyone else's position up by one
  db.prepare("UPDATE signups SET position = position - 1 WHERE event_id = ? AND status = 'waitlist'")
    .run(eventId);

  // Email the promoted person their confirmation
  const event = db.prepare('SELECT * FROM events WHERE id = ?').get(eventId);
  await sendPromoted({ to: next.email, name: next.name, event, cancelToken: newToken });
}

// WHAT I TOOK FROM IT

Why I Built It

The honest answer is that I was annoyed. I was doing the same manual work every week for every event, and it was obvious that a computer should be doing all of it. The system wasn't failing — it just had no reason to be as inefficient as it was except that nobody had sat down to replace it yet.

Building it turned out to be a genuine full-stack project in a way that portfolio exercises rarely are — there were real constraints, real users, and real edge cases. The late-cancel grace period for people promoted from the waitlist within 24 hours is a good example: that rule doesn't appear in any tutorial, but it's obviously the right call, and I only thought of it because I was building for a real situation with real people in it.

It also pushed my Node.js and SQL further than anything I'd built before. Getting waitlist position renumbering right, making cancellations work through email tokens without requiring a login, handling the weighted raffle draw, and building a multi-role auth system all took genuine problem-solving — not just following a pattern.

The bigger lesson was about initiative. The problem had existed for a long time before I got there. The fix wasn't complicated once someone decided to build it. Sometimes the most useful thing you can do is notice the thing nobody assigned.
