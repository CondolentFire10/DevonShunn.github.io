// THE STORY

Seven Websites to Get to One Good One

The first website I made was terrible. I knew nothing about HTML, so I found a template, changed the text, called it mine, and moved on. The second one was better in the way that something slightly less bad is better. By the third and fourth iterations I was actually learning — understanding why things worked, not just copying patterns that seemed to produce results.

The pattern across all six previous versions was the same: I'd build something that looked decent enough to publish, sit with it for a few months, and then find it embarrassing. Not because my taste had gotten worse — because every version taught me enough to see the problems the previous one couldn't. That's the thing about learning a skill in public. You get better fast, and your old work gets worse at the same rate.

This version — number seven — is the first one where I sat down with a clear visual direction, a coherent color system, real animations, and enough CSS and JavaScript knowledge to actually execute what I had in my head rather than compromise it into whatever I could get working. It's also, by a significant margin, the most time I've ever put into a personal website. Every page was thought about deliberately. Nothing is there because it was easy.

// THE ITERATION HISTORY

Six Versions That Led Here

Every version was the best website I knew how to make at the time. Every version got replaced because I learned enough to see why it wasn't good enough anymore.

VERSION 01

The Template Era

A downloaded template with the name and text swapped out. No real HTML knowledge — just enough confidence to change things without understanding why they worked. Published, linked it to a few people, immediately regretted most of the color choices.

VERSION 02

Learning by Copying

Wrote actual HTML this time — mostly by right-clicking on sites I liked and reading the source. Didn't understand the cascade in CSS yet, which meant a lot of !important hacks holding things together. Functional. Not pretty.

VERSION 03

The First Real Layout

Started understanding flexbox properly. First version with a real navigation, a consistent color palette, and hover states that weren't embarrassing. Still looked like a generic portfolio site — the kind you'd find on a "top 10 portfolio templates" list — but the HTML was actually clean.

VERSION 04

Going Dark

First dark-theme version. Introduced CSS variables for the first time so the color system was actually consistent rather than a collection of hardcoded hex values scattered across a thousand lines of CSS. First version that felt like it had a real aesthetic direction — even if it was "dark mode with blue accents," which is extremely not unique.

VERSION 05

JavaScript Enters the Picture

First version with real JavaScript — scroll-triggered animations using IntersectionObserver, a working mobile hamburger menu, and some basic interactive elements. The JS was messy and the animations were too aggressive, but it proved the concept. Also first version where the mobile experience was considered at all rather than just broken.

VERSION 06

Almost There

The predecessor to this one. Good structure, decent animations, clean CSS. But the design lacked identity — it could have been anyone's portfolio. No per-page theming, no real visual storytelling, nothing that felt specifically like it belonged to the projects it was supposed to represent. Close, but generic in a way that felt worse the longer it sat there.

You Are Here
VERSION 07 — THIS ONE

The One That Got the Most Work

Built with a clear visual language from the start — dark base, glass morphism cards, per-page accent colors that each feel distinct, scroll-driven animations, and project pages that are genuinely designed around their content rather than dropped into a shared template. The first version where I used AI assistance (Claude) as a collaborator — not to write the code for me, but to help execute the vision faster and work through the parts where my CSS knowledge still had gaps. The most honest, most complete representation of the work.

// HOW I LEARNED HTML

No Course. Just Building.

There was no bootcamp, no structured curriculum, no certificate at the end. Every step was driven by wanting to make something specific and not knowing how yet. That turns out to be a reasonably effective way to learn.

01

View Source Was the First Teacher

Right-clicking on websites I liked and reading the source code was how it started. Not to copy directly, but to understand what produced the visual I was looking at. Working backward from a result to the code that made it is a surprisingly fast way to build pattern recognition — you see the same properties showing up in the same contexts over and over, and it starts clicking.

View Source Pattern Recognition
02

Breaking Things on Purpose

Once I had something working, the fastest way to understand it was to break it deliberately — change a value, remove a property, add something unexpected — and watch what happened. CSS in particular rewards this approach because the feedback is instant and visual. You develop an intuition for what properties do to layout that no amount of reading documentation builds as quickly.

Experimentation Instant Feedback
03

Specific Problems, Specific Solutions

Learning flexbox happened because I needed to center something and floats were making me miserable. Learning CSS variables happened because I got tired of changing hex values in forty different places. Every major concept was learned because there was a concrete problem it solved — which meant by the time I understood it, I already knew exactly when and why to use it.

Problem-Driven Applied Context
04

AI as a Collaborator on Version 7

This version was the first time I used Claude as part of the build process. Not as a shortcut — I came in with the design direction, the color system, the structure of each page, and the specific effects I wanted. Claude helped execute the CSS for the parts I knew what I wanted but didn't know the exact syntax for, and helped debug the JavaScript in a few places where my logic had gaps. It's the fastest I've ever gone from concept to working page — and the most I've learned about CSS in a single build in the process.

// WHAT MADE THIS ONE DIFFERENT

What Version 7 Got Right

Six previous versions all had something wrong with them that became obvious over time. This version tried to fix every one of those things at once.

01

Per-Page Identity

Every project page has its own accent color, its own hero atmosphere, its own personality — cyan for the DeLorean, orange-amber for 3D printing, dark blue for BD-1, green for the railroad. The site has a consistent system but each page feels like it belongs to the project it's describing.

02

Actually Writing the Content

Previous versions had placeholder text that never got filled in properly. This version has real writing on every page — the stories behind the projects, what I actually learned, what went wrong, what I'd do differently. The content is as designed as the layout.

03

Scroll-Driven Animation Done Right

Earlier versions had animations that were either too aggressive (everything flying in all at once) or too subtle (no one noticed them). This version uses a single, consistent fade-up reveal triggered by IntersectionObserver — present enough to feel intentional, subtle enough to not distract from the content.

04

The Navigation That Actually Works

The dropdown navigation went through multiple revisions across earlier versions — gaps that closed it before you could click, timing that felt wrong, mobile states that broke. This version has a dropdown that works on hover desktop and tap mobile, stays open when you move the mouse into it, and doesn't fight you.

05

Glass Morphism That Doesn't Look Like a Template

Backdrop-filter glass effects are overused in portfolio sites — most of them look identical. This version uses it selectively, on elements where it adds depth rather than everywhere it can be applied. The difference between using a technique and using it well is restraint.

06

Hero Sections That Earn Their Height

A full-viewport hero section only works if there's something worth showing in it. This version's heroes have animated grids, ambient glows, particle systems, and background images that set the atmosphere for each project before a word is read. The space is justified rather than just spacious.

// A TASTE OF THE CSS

How the Glass Cards Work

The frosted glass card effect used throughout the site — deceptively simple once you understand what each property is doing.

glass-card.css 
/* Frosted glass card — the core pattern used site-wide */

.glass-card {
  background:    rgba(255, 255, 255, 0.06);
  border:        1px solid rgba(255, 255, 255, 0.12);
  border-radius: 16px;
  backdrop-filter: blur(14px);
  -webkit-backdrop-filter: blur(14px);
  transition:    border-color 0.3s ease, background 0.3s ease;
}

.glass-card:hover {
  background:    rgba(255, 255, 255, 0.09);
  border-color: rgba(232, 121, 249, 0.25); /* accent color */
}

/* The accent bar on the left edge of stat cards */
.stat-card::before {
  content:    '';
  position:   absolute;
  top:        0;
  left:       0;
  width:      3px;
  height:     100%;
  background: linear-gradient(to bottom, var(--accent), var(--purple));
  border-radius: 3px 0 0 3px;
}

The key insight was that backdrop-filter only works visually if the background has some transparency. Too much opacity and you lose the effect. Too little and it looks washed out. rgba(255,255,255,0.06) is the sweet spot on a very dark background.

// HTML/CSS TODAY

Where the Skills Are Now

Building this site was the most accelerated HTML and CSS education I've had. Six previous versions built knowledge incrementally — this version forced me to use almost everything at once, and working with AI assistance meant that when I hit a CSS gap, the explanation came alongside the fix.

The areas where I still have the most to learn are the advanced CSS stuff — container queries, the newer logical properties, CSS animations beyond the basics, and the parts of JavaScript that go beyond DOM manipulation and event listeners. This site pushed those limits pretty hard, which means the next version will start from a considerably higher floor.

There will be a version 8 eventually. Whatever it looks like, it'll have been made possible by every mistake made in versions 1 through 7.
